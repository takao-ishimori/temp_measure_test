# コード設計意図 解説

面接で「なぜこの書き方を？」と聞かれたときの回答集。

---

## 1. なぜ Python と C++ 両方書いたのか

**「同じ機能を両言語で実装して、精度とパフォーマンスを比較するためです」**

| | Python | C++ |
|---|--------|-----|
| DHT22読み取り失敗率 | やや多め | 少ない |
| 理由 | GILの影響でタイミングがずれる | マイクロ秒制御が正確 |
| HC-SR04 | 問題なし | 問題なし |
| 開発速度 | 速い | やや遅い |

→ **結果**：C++のほうがDHT22の読み取りが安定。でも両方動くので保守性と性能のトレードオフを実感できた。

---

## 2. なぜ C++ のコードは C に近い書き方なのか

**「wiringPiライブラリがC言語APIだからです。無理にC++風にせず、Arduino経験を活かせる書き方を意識しました」**

```cpp
// C++なのに printf / delay / digitalWrite → Arduino経験者なら読める
printf("気温: %.1f C\n", temperature);
delay(2000);
digitalWrite(PIN, HIGH);
```

→ 組み込みの現場はまだC言語が多いので、この書き方のほうが実践的。

---

## 3. なぜ MQTT を使ったのか

**「IoTの標準プロトコルで、疎結合な設計ができるからです」**

```
良くない設計:  DHT22.py → 直接SQLiteに書き込み
良い設計:      DHT22.py → MQTT → subscriber.py → SQLite
```

- 送信側と受信側がお互いを知らなくてよい
- 後から別のセンサーを追加しても、MQTTに流すだけで統合できる
- クラウド連携するときもMQTTブローカーの接続先を変えるだけ

---

## 4. なぜ SQLite なのか

**「ラズパイ1台で完結させるためです」**

MySQLやPostgreSQLを入れると：
- メモリ消費が増える
- 別途サーバープロセスが必要
- セットアップが面倒

SQLiteなら：
- Python/C++の標準ライブラリで使える（追加インストール不要）
- ファイル1つでDB完結
- 小〜中規模データなら十分な性能

---

## 5. なぜ Web ダッシュボードに Chart.js を使ったのか

**「追加インストール不要で、ブラウザさえあれば見られるからです」**

Grafanaと比較して：

| | Grafana | Chart.js |
|---|---------|----------|
| インストール | Grafanaサーバー必要 | 不要（ブラウザのみ） |
| 設定 | データソース設定要 | HTML+JSのみ |
| 可搬性 | 環境依存あり | どのPCでも開ける |
| 本番運用 | 強い | 簡易的 |

→ 個人プロジェクトの素早い可視化にはChart.jsが最適。本番ではGrafanaに載せ替え。

## 6. なぜ DHT22 の読み取りに `try-except` を多用しているのか

```python
try:
    temp = dht.temperature
except RuntimeError:
    pass  # 読み取り失敗は無視して次を試みる
```

DHT22は1-Wire通信の特性上、約2〜5%の確率で読み取り失敗が起きる。これはセンサーの仕様であり、異常ではない。
→ エラーを無視してリトライする設計が正しい。

---

## 7. なぜシステムモニターは `/proc` を直接読んでいるのか

```python
with open("/proc/stat", "r") as f:  # CPU使用率
with open("/proc/meminfo", "r") as f:  # メモリ使用率
```

**「外部ライブラリ不要で、Linuxの標準機能だけで情報が取れるからです」**

`psutil` などのライブラリを使うと依存が増える。`/proc` ファイルシステムはLinuxカーネルが提供する疑似ファイルで、余計なものなしでシステム情報が読める。軽量で確実。

---

## 8. なぜ DBマネージャーをモジュール分離したのか

```
database/db_manager.py      ← DB操作だけ
mqtt/mqtt_subscriber.py     ← MQTT受信 + DB保存
```

**「関心の分離（Separation of Concerns）のため」**

- DB操作のコードを1箇所にまとめることで、バグがあったときに修正箇所が明確
- MQTT受信側は「データを受け取ってdb_managerに渡す」だけの責務
- 別のプログラムからも `from db_manager import insert_reading` で再利用できる

---

## 9. なぜ C++ のDB保存で `roundf()` を使っているのか

```cpp
float t = std::roundf(temperature_c * 10.0f) / 10.0f;
```

**「浮動小数点の精度問題を防ぐため」**

`29.8` という値は float では `29.799999237060547` になる。そのままDBに入れると桁数が汚い。
→ 小数点以下1桁に丸めてから保存。SQLiteのREAL型でも見た目がきれいになる。

## 10. なぜ一括起動スクリプトを Python と Bash の両方で作ったのか

| | start_all.py | start_all_cpp.sh |
|---|-------------|-----------------|
| 起動するもの | Python版プロセス | C++ビルド＋起動 |
| 利点 | クロスプラットフォーム | 高速なセンサー読み取り |
| 使い分け | 開発・検証用 | 本番運用想定 |

Bash版はコンパイルから起動まで自動で行うので、`git clone` して `bash start_all_cpp.sh` で即運用可能。
